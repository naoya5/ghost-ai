# AI ワークフロールール (AI Workflow Rules)

## アプローチ (Approach)

[全体的な開発アプローチを記述する — 例: このプロジェクトは
spec 駆動ワークフローでインクリメンタルに構築する。
コンテキストファイルが、何を構築するか・どのように構築するか・
進捗の現状を定義する。常にこれらの spec に対して実装すること —
ゼロから挙動を推測したり創作したりしてはならない。]

## スコープルール (Scoping Rules)

- 一度に1つの機能ユニットだけに取り組む
- 大規模な投機的変更よりも、小さく検証可能な
  インクリメントを優先する
- 関連性のない複数のシステム境界を、1つの実装ステップで
  まとめて扱わない

## 作業を分割すべきタイミング (When to Split Work)

以下が組み合わさっている場合、実装ステップを分割すること:

- [関心事1 — 例: UI 変更とバックグラウンドタスク変更]
- [関心事2 — 例: 関連性のない複数の API ルート]
- [関心事3 — 例: コンテキストファイルで明確に定義されて
  いない挙動]

エンドツーエンドで素早く検証できない変更は、スコープが
広すぎる — 分割すること。

## 要件不足の取り扱い (Handling Missing Requirements)

- コンテキストファイルで定義されていないプロダクト挙動を
  創作しない
- 要件が曖昧な場合は、実装前に該当のコンテキストファイル
  内で解決する
- 要件が欠落している場合は、続行する前に
  `progress-tracker.md` に未解決の質問として追加する

## 保護されたファイル (Protected Files)

明示的な指示がない限り、以下を変更しないこと:

- [例: components/ui/* — 生成された UI ライブラリの
  コンポーネント]
- [例: サードパーティライブラリの内部]

## ドキュメントを実装と同期させる (Keeping Docs in Sync)

実装時に以下が変わったら、対応するコンテキストファイルを
更新すること:

- システムアーキテクチャまたは境界
- ストレージモデルの判断
- コーディング規約や基準
- 機能スコープ

## 次のユニットに移る前に (Before Moving to the Next Unit)

1. 現在のユニットが、定義されたスコープ内でエンドツーエンドに動作する
2. `architecture.md` で定義された不変条件が侵されていない
3. `progress-tracker.md` が完了済みの作業を反映している
4. `npm run build` が通る
