# アイデアからプロダクトへ: AI ドリブン開発者のためのプレイブック

### By JavaScript Mastery

---

> このガイドは、AI ドリブン開発のみで構築されたリアルタイム
> 共同編集型のシステム設計ツール「Ghost AI」のビルドと並行して
> 作成されたものです。本プレイブックの内容はすべて理論ではなく
> 実践で使われたものです。

このガイドを読み終える頃には、AI を使って白紙の状態から動く
プロダクトまで、どんなアイデアでも形にする方法が分かります。
プロンプトを上手く書く方法ではなく、プロンプトを書く前に
上手く考える方法です。

### 対象読者

- AI コーディングツールを試したが、破綻して苛立ったことがある開発者
- これから始めるところで、初日から正しい習慣を身につけたい開発者
- 本気で何かを作っていて、スケールするシステムが欲しい開発者

このメソドロジーを使うのに、長年の経験は要りません。
構築する前に考える意志があれば十分です。

### 得られるもの

- どんなプロジェクトでも AI ドリブン開発向けにセットアップする
  完全なシステム
- 自分のプロジェクト用に6つすべてのコンテキストファイルを生成する
  AI プロンプト
- すべてのコンテキストファイルの空テンプレート (記入式)
- 任意の機能をクリーンで構築可能な単位に分解する spec ファイルパターン
- 一貫してドリフトせずに AI と構築するための3プロンプトワークフロー
- AI が詰まったり間違え続けたりした時の明確な戦略

## パート1: 何かを構築する前に

ほとんどの開発者は AI コーディングツールを開いて、いきなり
タイピングし始めます。それが間違いです。

AI が生み出すものの品質は、完全にあなたが持ち込む思考の質に
よって決まります。明確なシステムは、クリーンで保守可能で
拡張可能なコードを生みます。曖昧なアイデアは、1時間ほどは
見栄えがするが週末までに崩壊するコードを生みます。

最初のプロンプトを書く前に、3つのことが必要です。
マインドセットの転換。AI プロジェクトが失敗する理由の理解。
そして正しい会話を行うこと。

---

### マインドセットの転換

AI で構築する上で最も重要な考え方は次のとおりです。

**あなたはアーキテクト。AI は実装エンジン。**

これが意味するのは、何を作るのか・どう組み合わさるのか・
ルールは何か・境界はどこか、といった思考はすべて常にあなたが
持ち続けるということです。AI はその思考を高速で実行する
だけで、置き換える存在ではありません。

AI に苦戦するほとんどの開発者は、思考をアウトソースしようと
しています。曖昧なゴールをエージェントに渡し、システム設計、
アーキテクチャ、コンポーネント境界、命名規約、データフローを
自分で考え出すことを期待します。たまに運良く成功します。
たいていは失敗します。失敗すると、開発者はツールを責めます。

AI で本気の物を作る開発者は、最初に思考を持ち込みます。
エージェントがコードに触れる前にシステムを設計します。
最初のプロンプトを送る前にルールを定義します。そしてその後、
人間1人では追いつけない速度でそのシステムを実行するために
AI を使います。

「AI、考えてくれ」から「AI、私が既に考えたことを実行してくれ」
への転換、それがすべてです。

---

### AI 支援プロジェクトのほとんどが失敗する理由

失敗モードは2つあります。両方を理解すれば、何週間もの
苛立ちを節約できます。

**失敗モード1: バイブコーディングの崩壊**

エージェントを開いて、欲しいものをざっくり説明し、走らせる。
最初の1時間は最高の気分です。コードがすごい速さで揃って
いきます。機能が次々と現れます。

そこに新しい何かを足そうとすると、何かが壊れる。直すと
別の何かが壊れる。エージェントは以前の判断と矛盾し始めます。
コードベースがあなたと戦い始めます。機能を作る時間より
AI のアウトプットをほどく時間のほうが長くなります。

これはエージェントに土台がなかったから起こります。
下した判断はすべて推測でした。そして推測は積み重なります。

**失敗モード2: 機能ドリフト**

最初のビルドはうまくいった。すべて動いた。2週間後に戻って
新しい機能を追加しようとすると、エージェントは何もかも
忘れています。アーキテクチャ上の判断、命名規約、ある
ロジックがなぜそう書かれたか。壊れていないものを「修正」
し、意図的に設定したパターンを上書きします。

これは AI エージェントがセッション間で記憶を持たないから
起こります。各セッション開始時に読むべき文書化された
システムが無ければ、エージェントは毎回ゼロから始めます。

両方の失敗モードの根本原因は同じです。開発者はエージェントに
作業の枠組みとなるシステムを与えなかった。解決策は、コードに
取り掛かる前にそのシステムを構築することです。

---

### コードの前にする会話

AI コーディングツールを開く前に、会話をしてください。

コーディングエージェントとではなく、プランニング AI と。
Claude、ChatGPT、Gemini、好きなものを使いましょう。
ゴールは声に出して考え、AI に考えを揺さぶってもらいながら
システムを明確化することです。

この会話こそが本当の作業です。シニアエンジニアが構築前に
頭の中やホワイトボードでやることを AI で外在化させ、
仮定を崩し、より速く行うのです。

その会話で扱うべきこと:

- これは実際に何をする物なのか?
- 誰が使い、何を必要としているか?
- 中心となるユーザーフローは最初から最後までどう流れるか?
- 技術的に最も複雑なのはどこか?
- 何が悪く転びうるか?
- どんな技術選択をするか、なぜか?
- 明確にスコープ外なのは何か?

急がないこと。この会話の中でシステムが頭の中で明確になればなる
ほど、その後すべてが良くなります。

---

### AI とアイデアを議論する方法

どんなプロジェクトでも、この会話を始めるためのプロンプト例:

```
私はあるアプリケーションのアイデアを持っています。
構築を始める前に、考え抜くのを手伝ってほしい。
こちらがアイデアです: [2-3文でアイデアを記述]
以下を明確化するために、一度に1つずつ質問してください:

- コアとなるユーザーフロー
- 技術的に最も複雑な部分
- データとストレージの要件
- 認証とアクセスモデル
- スコープ内とスコープ外
- 技術選択とその理由

私の回答が曖昧なときや、潜在的な問題が見えるときは反論してください。
コードを書く前にシステムについて明快に考えるのを手伝ってください。
```

迷わずすべての質問に明確に答えられるようになるまで、
会話を続けます。そこまで来たら、書き出す準備ができたサインです。

---

### コーディングツールを開く前に答えるべき質問

コーディングエージェントを開く前に、以下のすべてに答えられる
ようにしておくこと:

**プロダクト**

- このアプリは1文で何をするか?
- 主要ユーザーは誰で、コアニーズは何か?
- サインアップから本質的な価値到達までのステップ毎のフローは?
- 最初のバージョンで最も重要な3つの機能は?
- 明確にスコープ外は何か?

**技術**

- フルの技術スタックは何で、各選択の理由は?
- データはどこに置くか — DB、ファイルストレージ、キャッシュ?
- 認証はどのように動くか?
- システム境界はどこか — どのフォルダがどの責務を持つか?
- コードベースが絶対に侵してはならないルールは?

**デザイン**

- ビジュアル言語は — カラー、タイポグラフィ、スペーシング?
- どの UI コンポーネントライブラリを使うか?
- レイアウトを大まかにどう構成するか?

**プロセス**

- 主要機能を構築可能な単位にどう分解するか?
- それらをどの順番で構築すべきか?
- 各単位の「完了」とはどう見えるか?

これらすべてに答えられるなら、構築準備完了。答えられない
なら、答えられるまでプランニング AI と会話を続けてください。

## パート2: 基盤を構築する

頭の中でシステムが明確になったら、それを書き出します。

そこから生まれるのが Six-File Context System(6ファイル
コンテキストシステム)です。白紙のページに座ってドキュメントを
書こうとするのではなく、先ほどのアーキテクチャ会話の出力を、
プロジェクトのライフサイクル全体に同行する一連の文書群として
整理する、というやり方です。

これら6つのファイルは、コーディングエージェントが何かをする前に
読み込むものです。これがエージェントがあなたのプロジェクトを
知る方法であり、すべてのセッション、すべての機能、すべての
ビルド単位において一貫性を保つ方法です。これがなければ、
エージェントは推測します。あれば、エージェントは実行します。

---

### Six-File Context System とは

`context` または `docs` と名付けた1つのフォルダに6つのファイル。

```
context/
├── project-overview.md
├── architecture.md
├── code-standards.md
├── ai-workflow-rules.md
├── ui-context.md
└── progress-tracker.md
```

各ファイルには特定の役割があります。これらを合わせれば、
コーディングエージェントが1行のコードも書く前に、プロジェクトを
理解するために必要なすべてが揃います。

すべてのプロジェクトでこの正確な構造に従う必要はありません。
小さな週末プロジェクトなら4ファイルで十分かも。複雑なマルチ
チームの SaaS なら8ファイル必要かも。変わらないのは原則です —
エージェントが何かを構築する前に、何を作っているのか、どう
組み合わさるのか、ルールは何か、現在の状況はどうかを知って
いる必要がある、ということ。

このダウンロードに含まれる Ghost AI のファイル群は、現実世界の
完全な例です。実際のものがどう見えるか確かめるには、本セクションと
並行してそちらも読んでください。

### 各ファイルを AI で生成する方法

これらのファイルをゼロから書く必要はありません。**生成**します。

パート1で行った会話の出力を、各ファイルの入力として使います。
ファイルごとに、プランニング AI を開いて、後述のプロンプトを
使ってください。AI に下書きさせ、レビューし、プロジェクトを
正確に表現するまで磨いてください。

ゴールは「正確さ」であって「完璧さ」ではありません。これらの
ファイルはプロジェクトとともに進化します。今分かっていることから
始めて、判断が下されるたびに更新してください。

---

#### project-overview.md

**役割**

何を構築しているのか、なぜ構築しているのかを定義する。
ゴール、コアユーザーフロー、機能、スコープ境界、成功基準。

**重要な理由**

エージェントが spec が曖昧なときに「意図」を理解するために
読むファイル。これがないと、エージェントは仮定でギャップを
埋めます。これがあれば、曖昧な要件は推測ではなく定義済みの
プロダクトビジョンに照らして解決されます。

スコープ外セクションは特に重要です。何を構築しないかを
明示的にリストすることで、まだ必要のない機能のために依存を
増やしたりコードを書いたりしないようエージェントに伝えます。
スコープクリープはプロジェクトを殺します。このファイルは
エージェントがそれに加担しないようにします。

**🤖 生成用 AI プロンプト**

```
私のプロジェクトについての会話を踏まえ、project-overview.md
を書く手伝いをしてください。
含めるべき項目:

- アプリが何をするかの1段落の概要
- ゴールの番号付きリスト
- 開始から完結までステップ毎のコアユーザーフロー
- カテゴリ別に分けた機能セクション
- 構築する内容のスコープ内セクション
- 構築しない内容のスコープ外セクション
- 「完了」を定義する成功基準セクション

私のプロジェクト: [プランニング会話で出てきたアイデアと
主要な判断を貼り付け]
プレーンな Markdown で書いてください。具体的かつ明快に。
曖昧な表現を避けること。
```

**良いアウトプットとは**

ゴールは願望ではなく計測可能。ユーザーフローはステップ毎に
ギャップなし。スコープ外リストは明示的かつ詳細。成功基準は
「見栄えがいい」ではなく「サインイン済みのユーザーが
プロジェクトを作成して開ける」のように検証可能。

#### architecture.md

**役割**

完全な技術スタック、システム境界、ストレージモデル、認証モデル、
そしてコードベースが絶対に侵してはならない不変条件 (invariants)
を定義する。

**重要な理由**

このシステムで最も重要なファイル。これがないと、エージェントは
それらしく見えるアーキテクチャ判断を下し、徐々にコードベースを
腐食させます。物を間違ったレイヤーに置き、所有権チェックを
飛ばし、誤ったツールに手を伸ばします。

不変条件セクションは、良いアーキテクチャファイルと優れた
アーキテクチャファイルを分けるものです。不変条件は、システムが
絶対に侵してはならないルール — 例えば「リクエストハンドラは
長時間 AI 処理を実行しない」「認証はすべてのミューテーション境界で
強制される」など。不変条件がないと違反は静かに起こります。
あれば、違反はすぐに見えるようになります。

**🤖 生成用 AI プロンプト**

```
私のプロジェクト用に architecture.md を書く手伝いをしてください。
含めるべき項目:

- 各レイヤー、技術、その役割を表にしたスタック表
- システム境界 — どのフォルダがどの責務を持つか
- ストレージモデル — データベース vs ファイルストレージ vs キャッシュに
  何を入れるか
- 認証とアクセスモデル — 認証と所有権がどう機能するか
- 関連する場合の AI またはバックグラウンドタスクモデル
- 不変条件 — コードベースが絶対に侵してはならないルール

私のスタックと主要判断: [プランニング会話で出てきた
技術選択とアーキテクチャ判断を貼り付け]
適宜表を含めたプレーンな Markdown で書いてください。
具体的に。不変条件セクションには少なくとも4つのルールを
含めること。
```

**良いアウトプットとは**

スタック表はすべての技術について明確な役割が記述され完全。
システム境界は正確なフォルダ名と責務を指定する。ストレージ
モデルは明示的で、どこに何が住むか曖昧さがない。不変条件は
ガイドラインではなくルールとして記述される。

#### code-standards.md

**役割**

TypeScript の規約、フレームワークのパターン、API ルートの
構造、ファイル構成、スタイリングルールを定義する。

**重要な理由**

コーディング規約がないと、エージェントはドリフトします。
ユニット5で使うパターンが、ユニット20のパターンと違って
見える。型の扱いが一貫しない。コンポーネントが機能毎に
別々の構造で組まれる。個別には小さな差ですが、合わさると
読みづらく、拡張しづらく、引き継ぎづらいコードベースを
作ります。

#### ai-workflow-rules.md

**役割**

エージェントが構築中にどう振る舞うべきかを定義する —
スコープのルール、作業を分割するタイミング、不足要件の
扱い方、次に進む前の検証。

**重要な理由**

このファイルがエージェントを規律ある存在にします。これがないと、
エージェントは広く取りすぎ、関連のない関心事を1つのプロンプトに
混ぜ、不足要件について仮定を置き、検証をスキップします。
あれば、エージェントは自分のレーンに留まり、spec が言うことを
そのとおりに実行します — それ以上はしません。

**🤖 生成用 AI プロンプト**

```
私のプロジェクト用に ai-workflow-rules.md を書く手伝いを
してください。
このプロジェクトを構築する間に AI コーディングエージェントが
どう振る舞うべきかを定義します。含めるべき項目:

全体的なアプローチ (spec 駆動、インクリメンタル)
スコープルール (一度に1ユニット、推測的な変更禁止)
作業をより小さなステップに分割するタイミング
不足または曖昧な要件の扱い方
明示的指示なしに変更してはならないファイル
(例: 生成された UI ライブラリのコンポーネント)
ドキュメントを実装と同期させる方法
次のユニットに移る前の検証チェックリスト

エージェントへの直接の指示として書いてください。ガイドライン
ではなくルールとして。命令形を使うこと。
```

**良いアウトプットとは**

提案ではなく直接の命令として書かれている。「スコープを小さく
保つよう努めて」ではなく「一度に1つの機能ユニットだけに
取り組む」。不足要件のセクションは具体的で、何かが未定義の
ときに何をすべきかをエージェントに正確に伝える — 推測の余地を
残さない。

---

#### ui-context.md

**役割**

ビジュアル言語を定義する — カラートークン、タイポグラフィ、
ボーダー半径スケール、コンポーネントライブラリの規約、
レイアウトパターン、アイコンの使い方。

**重要な理由**

このファイルがないと、エージェントは視覚的判断のすべてを
推測します。トークンではなく生のカラー値を使う。一貫性のない
ボーダー半径を当てる。機能間でコンポーネントパターンが
混じる。結果は10人の異なる開発者が作ったように見える UI です。

このファイルがあれば、エージェントは視覚的判断を一切しません。
spec を読むだけです。

**作成方法**

このファイルは他とは異なり、プランニング会話だけからは生成
しません。先にデザインしてから文書化します。

会話から始める:

```
[アプリの説明] を作っています。これに合うカラートークン
システムを設計するのを手伝ってください。
欲しい雰囲気: [雰囲気を記述 — ダーク/ライト、
テクニカル/フレンドリー、ミニマル/リッチなど]
生成してほしいもの:

- セマンティックなトークン名と hex 値が付いた完全な
  カラーパレット
- タイポグラフィ推奨
- ボーダー半径スケール
- 関連すれば AI またはアクセントカラーバリアント

ui-context.md に貼り付けられる Markdown 表として
出力してください。
```

その後、生成されたものの上にコンポーネントライブラリ規約、
レイアウトパターン、アイコンライブラリを追加します。

**良いアウトプットとは**

すべての色は名前付きトークン、生の hex 値ではない。
エージェントが色を発明する必要が一切ないこと — 起こり得る
すべてのカラー判断がこのファイル内にある。レイアウトパターンは
アプリの実際の構造を記述する — サイドバーの挙動、モーダルの
パターン、ナビバーの構造など。

---

#### progress-tracker.md

**役割**

現フェーズ、完了済み、進行中、次に来るもの、未解決の質問、
アーキテクチャ判断、セッションメモを追跡する。

**重要な理由**

ビルド全体で常に変わる唯一のファイル。そして長いビルドでは
最も重要なファイル。

AI エージェントはセッション間で記憶を持ちません。新しい
セッションを開くたびに、エージェントはゼロから始まります。
進捗トラッカーは、1つのプロンプトで完全なコンテキストを
復元する方法です。1つの指示「エントリーファイルを読んで
再開」と打てば、エージェントはプロジェクトがどこに立って
いて、どんな判断が下されてきて、次に何が来るかを正確に
把握します。

このファイルがないと、毎セッションの最初の15分を、自分の
プロジェクトをエージェントに再説明することに費やします。
あれば、数秒で動き出します。

**セットアップ方法**

他5ファイルと違い、進捗トラッカーは空で始めます。
このテンプレートをコピーし、最初の2セクションを埋めます:

```markdown
# Progress Tracker

意味のある実装変更があるたびに、このファイルを更新する。

## Current Phase

- [フェーズ名]

## Current Goal

- [今構築しているもの]

## Completed

- まだなし

## In Progress

- まだなし

## Next Up

- [最初に構築するユニット]

## Open Questions

- [未解決の判断事項]

## Architecture Decisions

- [システム設計に影響する判断]

## Session Notes

- [次のセッションで再開するために必要なコンテキスト]
```

エージェントは各ユニット後にこのファイルを更新します。
あなたはアーキテクチャ判断を下したり未解決の質問を解決したり
したときに更新します。プロジェクト終了時には、下されたすべての
判断とその理由の完全な記録になります。

---

### CLAUDE.md/AGENTS.md — エントリーポイント

6ファイルが揃ったら、もう1つファイルが必要です — コーディング
エージェントが何かをする前にどこを見るべきかを伝える
エントリーポイントです。

主要なコーディングエージェントには必ずあり、名前が違うだけです。
Claude Code は `CLAUDE.md` を読みます。Codex と GitHub Copilot は
`AGENTS.md` を使います — これは最も一般的な慣例になりつつあり、
Next.js が新しいプロジェクトを初期化するときに自動生成するように
なりました。Cursor は `.cursorrules` を使います。Windsurf は
独自の慣例を持ちます。

自分のコーディングエージェントに合うファイル名を使ってください。
内容と目的は同じ — ルートに1ファイル、エージェントが各セッションで
最初に読むもの。

テンプレート:

```markdown
## Application Building Context

実装やアーキテクチャ判断を行う前に、以下のファイルを順番に
読み込むこと:

1. `context/project-overview.md` — プロダクトの定義、
   ゴール、機能、スコープ
2. `context/architecture.md` — システム構造、
   境界、ストレージモデル、不変条件
3. `context/ui-context.md` — テーマ、カラー、タイポグラフィ、
   およびコンポーネント規約
4. `context/code-standards.md` — 実装ルールと
   コーディング規約
5. `context/ai-workflow-rules.md` — 開発ワークフロー、
   スコープルール、デリバリーアプローチ
6. `context/progress-tracker.md` — 現フェーズ、
   完了済み作業、未解決質問、次のステップ

意味のある実装変更があるたびに `context/progress-tracker.md`
を更新する。

実装によってコンテキストファイルに記載されたアーキテクチャ、
スコープ、または基準が変わる場合は、続行する前に該当ファイルを
更新する。
```

このファイルが、システム全体を機能させる指示です。これがないと、
エージェントはコンテキストファイルをスキップして仮定を置き、
あなたが定義したアーキテクチャからドリフトしかねません。
あれば、すべてのセッションが同じ形で始まります — 完全な
コンテキスト、定義済みのシステム、明確なルールで。

### 同梱テンプレートの使い方

このダウンロードの `templates/` フォルダには、すべてのコンテキスト
ファイルの空版が含まれています — 自分のプロジェクト用に記入できる
状態です。

使い方:

1. `templates/` フォルダ全体をプロジェクトルートにコピーし、
   `context/` にリネーム
2. 各ファイルを開き、本ガイドの AI プロンプトを使って自分の
   プロジェクト固有の内容を生成
3. 各ファイルが自分のシステムを正確に表現するまでレビュー・調整
4. 提供されたテンプレートを使って `CLAUDE.md` をプロジェクト
   ルートに追加

テンプレートが構造を、パート2の AI プロンプトが内容を提供します。
プランニング会話が、AI が正しく埋めるために必要なすべてを
与えます。

完成した本番品質のコンテキストファイルが実際にどう見えるかを
見たい場合、Ghost AI の完全なコンテキストファイルが無料の
ビデオキットに含まれています。リンクは動画概要欄に。

## パート3: ビルドを分解する

コンテキストファイルが揃った。エージェントはあなたのプロジェクトを
理解した。

最初のプロンプトを書く前に、もう1つだけ答えるべき質問が
あります — 正確に何を、どんな順番で構築し、各ピースが完了したと
どうやって判断するか。

これが **spec 駆動開発** です。

エージェントを開いて思いつくままに機能ごとにプロンプトする
のではなく、コードを1行も書く前に全ビルドを上流で計画し、
具体的でスコープが明確で検証可能な単位に分解します。各単位には
spec が、各 spec には「完了」の定義があります。

ほとんどの開発者がここを飛ばします。コンテキストファイルが
できた途端にプロンプトを始めます。結果は、最初は速く感じるが
中盤からカオスになるビルド — ピースをどう組み合わせるかの
計画がなかったから。

Spec 駆動開発はそれを直します。すべての単位は計画される。
すべての依存は順序付けられる。すべての機能はエージェントが
触れる前に明確な終点を持つ。ビルドは最初のプロンプトから最後の
プロンプトまで予測可能でいられます。

---

### プロジェクトを単位 (units) に分解する方法

単位とは、単一でスコープが明確で検証可能な作業のかたまり。
1つの集中セッションで作れるくらい小さく、「完了」がどう
見えるか分かるくらい具体的なもの。

「ダッシュボードを作る」ではない。それはフェーズ、単位ではない。

単位はこんな感じ: 「My Projects と Shared タブを持つ
プロジェクトサイドバー、空のプレースホルダー状態、開閉動作を
構築する。API 呼び出しはまだなし。」

これは具体的、スコープが明確、終点がはっきりしている。

**良い単位のルール:**

- 目に見えて検証可能な結果を1つ生む
- 1つのシステム境界の中に留まる — UI 変更とデータベース変更と
  バックグラウンドタスクを1つの単位に混ぜない
- 完了とみなす前に真でなければならない条件のチェックリストを持つ
- 別の単位に属する判断を下さずに、1つの集中セッションで
  構築できる

**AI を使った分解方法:**

```
[アプリ名] を作っています。コンテキストファイルが完全な
アーキテクチャと機能セットを定義しています。
以下のルールに従ってビルド全体を単位に分解するのを手伝って
ください:

- 各単位は目に見える結果を1つ生む
- 各単位は1つのシステム境界の中に留まる
- 依存はジャストインタイムで導入する。必要になる前に何かを
  インストールしたり構築したりしない
- 同じセッションで常に一緒に行われる単位は1つに統合する
- 単独では目に見える結果がない単位は隣接する単位に統合する

機能リストはこちら: [project-overview.md から機能を貼り付け]
スタックはこちら: [architecture.md からスタックを貼り付け]
ビルド順の番号付きリストとして出力してください。各単位について、
単位番号、単位名、何を構築するか、先に存在すべき依存を含めて
ください。
```

出力を慎重にレビューしてください。順序が意味をなさないところは
並び替え、単独で立てないほど小さい単位は統合し、1つで多くを
やろうとしすぎている単位は分割してください。

**結果があなたのビルド計画 (build plan)。** これを
`context/specs/00-build-plan.md` として保存します。最初の
プロンプトを送る前にシステム全体が設計された証拠であり —
デモや動画の冒頭で信頼性を確立するために見せるファイルです。

---

### Spec ファイルパターン

ビルド計画の各単位は、独自の spec ファイルを持ちます。
spec ファイルはその単位の完全な指示書 — 曖昧なプロンプトの
代わりです。

コーディングエージェントは spec ファイルを読み、書かれた
通りに実装します。それ以上でも以下でもない。

良く書かれた spec ファイルには5つのセクションがあります:

**1. ゴール (Goal)**
1〜2文。完了時に何を生む単位か?具体的に。

_悪い例:_ `"認証ページを作成する"`

_良い例:_ `"Clerk のコンポーネントを使って、デスクトップは
2パネルレイアウト、モバイルはフォームのみのサインインと
サインアップページを作成する。ルート保護は middleware.ts ではなく
proxy.ts を使う。"`

**2. デザイン (Design)**
この単位固有の視覚的・構造的決定。ui-context.md のトークンを
参照し、レイアウト挙動、コンポーネント選択、レスポンシブ要件を
記述する。これによってエージェントが視覚的な推測をしなくなる。

**3. 実装 (Implementation)**
サブセクションに分割 — コンポーネントまたはシステム境界毎に1つ。
各セクションでエージェントに何を、どう構築するかを正確に伝える。
完了がどう見えるかについて曖昧さがないだけの詳細を含める。

**4. 依存 (Dependencies)**
この単位が必要とする、まだインストールされていないパッケージ。
明示的にリスト化する。エージェントに必要なものを推測させない。

**5. 検証チェックリスト (Verification checklist)**
この単位が完了する前に真でなければならない具体的条件のリスト。
エージェントもこれに対してチェック、移る前にあなたもこれに対して
チェック。

**Spec ファイルテンプレート:**

```markdown
# Unit NN: [機能名]

## Goal

この単位が生む具体的なアウトプットを1〜2文で。

## Design

この単位固有の視覚的・構造的決定。
関連箇所では ui-context.md のトークンを参照する。

## Implementation

### [コンポーネントまたはサブセクション名]

何を構築するかの詳細な記述。

### [次のサブセクション]

記述。

## Dependencies

- パッケージ名 (理由)

## Verify when done

- [ ] 条件1
- [ ] 条件2
- [ ] TypeScript エラーなし
- [ ] コンソールエラーなし
- [ ] モバイルとデスクトップでレスポンシブ
- [ ] npm run build が通る
```

### AI で機能 spec を生成する方法

spec ファイルをゼロから書く必要はありません。各機能を実装する
直前に生成します。

単位を構築する準備ができたら、まずプランニング AI と短い会話を
してください。共有するもの:

- コンテキストとして `project-overview.md` と `architecture.md`
- この単位で構築したいもの
- 適用される具体的な決定や制約

それから spec ファイルを書くよう依頼します:

```
プロジェクト概要: [project-overview.md を貼り付け]
アーキテクチャ: [architecture.md を貼り付け]
実装したいもの: [機能を記述]
以下の構造に従って、この機能の詳細な spec ファイルを
書いてください:

- ゴール (1〜2文、具体的)
- デザイン (視覚的・構造的決定)
- 実装 (サブセクションに分割)
- 依存 (インストールするパッケージ)
- 検証チェックリスト

可能な限り具体的に。不明な点があれば、spec を書く前に
質問してください。
```

または、何を構築したいか既に分かっている場合は、自分で
ざっくり書いて AI に整形してもらうだけでも OK です。

いずれにせよ、spec が詳細であるほど、コーディングエージェントの
出力は良くなります。曖昧な spec は曖昧なコードを生み、
詳細な spec はクリーンで予測可能な実装を生みます。

1つの機能が1つの spec で済むこともあれば、3つ必要な
こともあります。固定ルールではなく機能の複雑さに決めさせて
ください。

### ビルド計画 — 単位を正しく順序付ける

単位の順序は、多くの開発者が思っている以上に重要です。
順序を間違えると、まだ存在しない土台の上に機能を作ったり、
使う準備ができていない土台を作ったりすることになります。

**単位順序付けのルール:**

**依存が先。** 機能 B が機能 A を必要とするなら、A が先。
存在しないものの上に絶対に作らない。

**機能性よりセキュリティ。** 認証とアクセス制御は、それが
保護する機能より常に先。アクセス制御モデルが存在する前に
共同編集キャンバスを作るのは、無防備な土台の上に作ること。

**フロントエンド配線よりバックエンド先。** API ルートを
先に作り、それから UI を配線する。両方を1つの単位に組み合わせると、
エージェントが仮定を行う表面積が大きすぎる。

**実データより UI シェル先。** プレースホルダーデータで
コンポーネント構造を先に作り、それから実 API 呼び出しに繋ぐ。
データ層が存在する前に UI が動くことを検証できる。

**依存はジャストインタイムでインストール。** 実際の挙動を
解禁する単位が来てからインストールする。すべてを上流で
インストールしない — コンテキストにノイズを生み、エージェントが
まだ使うべきでないツールに手を伸ばす原因になる。

**ビルド順を検証する方法:**

単位リストを順に追って、各単位について問う: この単位が
依存するものは前の単位で既に存在するか?ノーなら並び替える。

また問う: 隣接する2つの単位が常に同じセッションで完了し、
その間に独立した結果がないか?イエスなら1つに統合する。

順序が正しければ、すべての単位が前の単位の上にきれいに
積み上がります。先送りを必要とする単位はなく、3単位後まで
動かないものを残す単位もない。

その予測可能性こそが、ビルドをカオスではなくスムーズに感じさせる
ものです。

## ワークフロー要約

コンテキストファイルが揃った。ビルド計画もできた。
最初の spec ファイルも書けた。

ここからのビルドの回し方:

**実装する1つのプロンプト:**

```
context/specs/NN-feature-name.md を読んでください。
context/progress-tracker.md を更新して進行中とマークしてください。
spec の通りに実装してください。
この単位のスコープを超えないでください。
```

**何かズレていたら修正する1つのプロンプト:**

```
[具体要素] が spec と一致していません。
期待: [spec が言うこと]。
現在: [作られたもの]。
これだけ修正してください。他は何も変えないでください。
```

**クローズする1つのプロンプト:**

```
実装が完了し検証済みです。
context/progress-tracker.md でユニット NN を完了とマークしてください。
ブランチ feat/NN-feature-name を GitHub にプッシュしてください。
```

このサイクルがビルド計画の各単位で繰り返されます。
コンテキストファイルがエージェントを一貫させ、spec ファイルが
各単位を集中させ、進捗トラッカーが各セッションをプロジェクトの
実際の状態に接地させます。

完全なワークフロー — AI のアウトプットをレビューする方法、
バグの扱い方、既存を壊さずに新機能を追加する方法、本番への
デプロイ — については、JavaScript Mastery YouTube チャンネルの
完全な Ghost AI ビルドを見てください。
